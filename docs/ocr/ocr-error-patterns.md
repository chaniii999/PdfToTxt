# OCR 오인식 패턴 분석 (실측 데이터 기반)

AI Agent가 OCR 개선 시 참고할 오인식 유형·패턴·대응 방안 문서.

---

## 1. 오인식 유형 분류

| 유형 | 설명 | 대응 난이도 |
|------|------|-------------|
| **유형 A** | 비슷한 글자끼리 헷갈림 (형태 유사) | 전처리·PSM 튜닝으로 완화 가능 |
| **유형 B** | 전혀 다른 한글이 영어로 표현됨 | kor+eng 혼용 시 Tesseract 근본 한계 |

---

## 2. 유형 A: 비슷한 글자끼리 헷갈림

## 3. 유형 B: 전혀 다른 한글이 영어로 표현됨

한글이 Latin 문자로 완전히 잘못 인식되는 패턴. kor+eng 혼용 시 Tesseract가 한글 자모를 Latin으로 읽는 경향.

## 5. 대응 방안 요약

### 유형 A (비슷한 글자 헷갈림)

| 방안 | 설명 | 구현 상태 |
|------|------|-----------|
| DPI 상향 | 350 DPI로 자모 경계 선명화 | ✅ 적용 |
| sharpen + CLAHE | `enhance_for_ocr` | ✅ 적용 |
| morphology close | preset C, 끊긴 획 연결 | ✅ 적용 |
| tessdata_best | 12MB kor, 4.7MB eng | ✅ 권장 |
| ㅇ/ㅁ, ㅈ/ㅅ 구분 | 전처리로 자모 경계 강화 | 🔄 검토 |

### 유형 B (한글→Latin 오인식)

| 방안 | 설명 | 제약 |
|------|------|------|
| kor 단독 OCR | 한글 우선 인식 | 영문→숫자 변환 부작용 (improvement-log) |
| kor+eng 유지 | 현재 전략 | Latin 오인식 근본 한계 |
| 후처리 치환 | 0!!→PII, ㄴㄴ→LLM 등 | 인식률 우선으로 제거됨. 패턴 확장 시 재검토 |
| LLM 후처리 | 요약 단계에서 문맥 보정 | 파이프라인 하류에서 처리 가능 |

### 권장 방향

1. **유형 A**: 전처리·PSM·tessdata 품질에 집중. ㅇ/ㅁ, ㅈ/ㅅ 등 혼동 자모에 대한 전처리 실험.
2. **유형 B**: Tesseract kor+eng 한계 인정. 요약·분석 단계에서 LLM이 문맥 기반 보정하거나, 도메인 특화 치환 테이블(의회문서용)을 선택적으로 적용 검토.
