# 한글→영문 오인식 원인 분석

멀쩡한 한글이 영문으로 인식되는 이유를 로직 관점에서 분석.

---

## 1. 오인식 패턴 (실측 데이터)

| 원본(한글) | 추출(오인식) | 유형 |
|------------|--------------|------|
| 기업 | JHA | 한글→영문 |
| 보호 | BSE | 한글→영문 |
| 자기 | DEE | 한글→영문 |
| 생성 | MM | 한글→영문 |
| 과정 | REHM | 한글→영문 |

| 보안 | 20 | 한글→숫자 |

---

## 2. 원인 분석

### 2.1 자모-알파벳 시각적 유사성

| 한글 | 유사 영문 | 비고 |
|------|-----------|------|
| ㅁ | M | 모양 유사 |
| ㅇ | O, 0 | 원형 |
| ㅂ | B | |
| ㅅ | S | |
| ㅎ | H | |
| ㅈ | J | |
| ㄱ | r, k | |
| ㅏ | a | |
| ㅓ | e | |
| ㅗ | o | |

kor+eng 동시 사용 시 Tesseract가 한글 영역을 영문으로 잘못 분류.

### 2.2 Tesseract 언어 모델

- `kor+eng`: kor가 먼저 나와 한글 우선이지만, LSTM이 혼합 문서에서 스크립트 판별을 잘못할 수 있음
- 비사전어(영문 오인식)에 대한 페널티가 기본값보다 낮으면 "JHA", "BSE" 등이 선택될 가능성

### 2.3 획 구분 부족

- 한글 자모는 획이 얇고, 이진화 후 끊기면 ㅇ/ㅁ, ㅈ/ㅅ 등이 혼동
- 선명도 부족 시 ㅂ↔B, ㅅ↔S 등 시각적 유사도 증가

---

## 3. 적용한 조치

| 조치 | 파일 | 효과 |
|------|------|------|
| Sharpen | preprocess_minimal | 자모 경계 선명화 |
| Morphology close | preprocess_minimal | 끊긴 획 연결 |
| language_model_penalty_non_dict_word=0.25 | pdf_ocr | 비사전어(영문 오인식) 페널티 상향 |
| language_model_penalty_non_freq_dict_word=0.2 | pdf_ocr | 비빈도어 페널티 |
| 후처리 패턴 | postprocess | JHA→기업, BSE→보호 등 교정 |

---

## 4. 한계

- Tesseract는 kor/eng를 별도 모델로 사용. 혼합 문서에서 스크립트 판별이 근본적 한계.
- 자모-알파벳 유사성은 훈련 데이터로만 완전 해결 어려움.
- 후처리 패턴은 실측 데이터 기반으로 확장 필요.
