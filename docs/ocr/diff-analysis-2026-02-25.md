## 2. 원인 분석

### 2.1 image_to_data level 미필터 (가장 유력)

`image_to_data`는 level 1~5(페이지/블록/단락/줄/단어)를 모두 반환한다.

- **현재**: level 구분 없이 모든 행 사용
- **문제**: level 5(단어)만 text가 제대로 채워지고, 상위 level은 빈 값·중복·잘못된 순서
- **결과**: 읽기 순서가 꼬이고, "=" 같은 노이즈 블록이 섞임

**조치**: `level == 5`인 행만 사용.

---

### 2.2 2단(2-column) 레이아웃

문서가 좌우 2단 구조인데, PSM 3(자동 레이아웃)이 열 구분을 잘못 인식할 수 있음.

- **가능성**: 왼쪽/오른쪽 열을 섞어 읽거나, 여백·헤더를 먼저 읽음
- **증거**: "마", "=H", "=" 등이 앞에 나오는 것은 상단 여백/헤더/구분선일 가능성

**조치 후보**:
- 2단 감지 후 이미지를 좌/우로 나누어 각각 OCR 후 병합
- 또는 PSM 6(단일 블록) + 열 분할

---

### 2.3 표선·구분선이 문자로 인식

"="가 반복되는 것은 표 경계, 구분선, 밑줄 등이 이진화 후 선으로 남아 "="로 인식되는 것으로 보임.

- **현재**: `remove_table_lines` 비활성화(문자 손상 우려)
- **문제**: 선이 그대로 남아 노이즈 블록 생성

**조치 후보**:
- 얇은 선만 제거하는 전처리(문자 획은 유지)
- 또는 후처리에서 단일 "=", "—" 블록 제거

---

### 2.4 전처리 누락 (ocr-settings.json 대비)

| 항목 | ocr-settings | 현재 preprocess_minimal |
|------|--------------|-------------------------|
| Gaussian Blur | 있음 | **없음** |
| Adaptive Threshold | block_size=11, C=2 | Otsu만 사용 |
| 10px 흰 테두리 | 있음 | **없음** |
| Deskew 최소 각도 | 0.5° | 1.0° |

Otsu만 사용하면 그림자·얇은 선에 과민 반응해 노이즈가 늘어날 수 있음.

---

### 2.5 Hybrid OCR 블록 병합 순서

`line_num` 기준으로만 병합하는데, PSM 3이 잘못된 block/line 순서를 주면 전체 읽기 순서가 틀어짐.

- level 5만 쓰면 개선 가능
- 2단 문서는 열 분할이 더 확실함

## 3. 우선순위별 조치

| 순위 | 조치 | 기대 효과 |
|------|------|-----------|
| 1 | **level == 5 필터** | 읽기 순서·중복·노이즈 블록 대폭 감소 |
| 2 | 10px 흰 테두리 추가 | 가장자리 텍스트 인식 안정화 |
| 3 | Gaussian Blur + Adaptive Threshold | 노이즈 감소, 영문 디테일 유지 |
| 4 | 단일 "=", "—" 블록 후처리 제거 | 표선 노이즈 제거 |
| 5 | 2단 감지·열 분할 | 2단 문서 읽기 순서 정확도 향상 |

---

## 4. 요약

- **핵심**: `image_to_data`에서 `level == 5`만 사용하지 않아, 상위 level의 잘못된/빈 블록이 섞여 순서와 품질이 크게 나빠짐.
- **부가**: 2단 레이아웃, 표선 노이즈, 전처리(블러·Adaptive·테두리) 미적용이 추가로 악화 요인.
